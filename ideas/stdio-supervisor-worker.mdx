# Stable stdio supervisor with disposable Payload workers

## Status

This is the strongest-isolation standalone design.

It keeps the MCP process and its stdin/stdout connection alive while replacing the process that
loads the Payload config. The process boundary is the important part: every successful reload gets
a genuinely new Node module graph instead of trying to invalidate an existing ESM graph.

## Why this option remains

This design satisfies the narrowed requirements:

- It does not depend on `notifications/tools/list_changed` or any equivalent prompt/resource event.
- It works with harnesses that initialize and list capabilities only once.
- It can keep serving the previous valid config while a replacement boots, which is at least as
  reliable as the current HTTP development path.
- It does not rely on query-string cache busting, `require.cache` deletion, or assumptions about
  which modules `tsx`, Node, Payload, adapters, or user code have cached.
- It remains a real standalone stdio server. A Next.js development server is not required.

## Architecture

```text
MCP harness
    │ one long-lived stdin/stdout connection
    ▼
stdio supervisor
    ├── fixed MCP surface
    ├── dependency watcher
    ├── active generation routing
    └── reload coordinator
             │ private IPC
             ▼
       Payload worker generation
       ├── fresh Node process
       ├── fresh tsx loader
       ├── fresh config import graph
       ├── fresh Payload instance
       └── current tools, prompts, and resources
```

The supervisor owns stdout for its entire lifetime. Workers must never inherit stdout because one
stray Payload log would corrupt the MCP stream. Worker logs and diagnostics go to stderr or over a
separate IPC channel.

## Fixed external MCP surface

The external surface cannot add or remove named tools after initialization if this must work in
every harness. A harness that neither handles list-change notifications nor reconnects has no way
to discover those changes.

Payload's built-in entity operations are already suitable for a fixed surface because operations
such as `findDocuments` accept a collection slug. The supervisor should expose a stable set along
these lines:

- `getConfigInfo`
- collection and global operations using `collectionSlug` or `globalSlug`
- schema inspection operations
- `listCustomTools` and `callCustomTool`
- `listPrompts` and `getPrompt`
- `listResources` and `readResource`

The exact names can follow the existing API, but their names and input schemas must not depend on
the loaded Payload config. The current worker resolves slugs, schemas, access, custom definitions,
and handlers at call time.

Config-defined native MCP items may still be exposed as a best-effort convenience for harnesses
that refresh their lists. They cannot be the only route to those items. The generic dispatch tools
are the compatibility contract for all harnesses.

## Worker lifecycle

The initial generation performs the same work as the current `runMcpStdio` implementation:

1. Resolve and import the Payload config.
2. redirect Payload logging away from stdout.
3. call `getPayload({ config, cron: false, disableOnInit: true })`.
4. resolve and authorize the MCP plugin config.
5. report `ready` only after it can serve calls.

On a relevant file change, the supervisor:

1. Starts generation `N + 1` in a new child process.
2. Leaves generation `N` active while the new config imports and Payload initializes.
3. Waits for a structured `ready` message containing the new dependency set and capability
   metadata.
4. Atomically changes the active-generation pointer.
5. Routes new calls to `N + 1`.
6. Allows calls already assigned to `N` to finish, up to a timeout.
7. asks `N` to run `payload.destroy()`, then terminates it if it does not exit.

Only one candidate generation should boot at a time. If another change arrives during startup,
mark the candidate stale and boot once more from the newest filesystem state.

## Failed reloads

A syntax error, invalid config, unavailable database, or failed plugin initialization must not
break the stdio connection.

The candidate reports a structured failure over IPC and exits. The supervisor keeps the last known
good generation active, writes the failure to stderr, and retries after the next relevant change.
If the first generation cannot boot, the supervisor should still complete the MCP handshake and
return a clear temporary-unavailable result from its fixed tools. This prevents harnesses with weak
process-restart behavior from becoming permanently wedged.

## Dependency tracking

Watching only `payload.config.ts` is insufficient because most config behavior lives in imported
collections, fields, hooks, access functions, plugins, and helper modules.

The worker should collect local modules loaded by the `tsx` import path and send their canonical
file paths to the supervisor. The supervisor watches:

- the resolved config file;
- every local imported dependency reported by the active and candidate generations;
- parent directories needed to detect atomic-save rename patterns.

It should ignore `node_modules`, package build output, generated Payload types, database files, and
other known write-heavy paths. Changes should be debounced and coalesced.

When a config edit adds a new import, the config itself triggers a reload and the new worker reports
the expanded graph. When an import is removed, the dependency set is replaced after the new worker
becomes active.

## Why not reload in the supervisor process

The module graph is larger than the config module. It can include Payload, plugin instances,
database adapters, generated modules, project packages, and arbitrary user dependencies. ESM has no
supported recursive cache eviction API. Adding a query to the root import does not guarantee fresh
transitive imports, and deleting `require.cache` does not clear ESM or loader-owned caches.

A child process gives each generation fresh:

- Node ESM and CommonJS caches;
- `tsx` loader state and namespaces;
- module-level Payload and plugin singletons;
- config closures and registered handlers;
- adapter state that may otherwise survive a partial reload.

Of the standalone options, this has the simplest freshness invariant: correctness does not depend
on knowing every cache that participated in the previous generation. The in-process HMR option can
also be viable, but only when a real module runner owns and invalidates the complete editable graph.

## Worker protocol

The private protocol can be a small request/response API over Node IPC or a private MCP connection.
It needs to carry:

- generation identity and readiness;
- imported dependency paths;
- current config/capability metadata;
- generic tool, prompt, and resource calls;
- cancellation and progress events;
- structured errors and shutdown acknowledgements.

If private MCP is used, the supervisor still needs an adapter so the public fixed surface can call
the worker's config-dependent named surface. `ServerContext` features such as cancellation,
progress, logging, and elicitation must be forwarded rather than silently discarded.

## Verification

The implementation needs real-process tests, not only mocked module imports:

- initialize once, edit an imported collection, and observe the new schema without reconnecting;
- add and remove a collection without relying on list-change support;
- change a custom tool implementation while keeping its name;
- add a custom tool and invoke it through the generic dispatcher;
- change a transitive helper module;
- make the next config invalid and verify the old generation still serves calls;
- fix the config and verify recovery on the same stdio session;
- edit repeatedly during a slow initialization and verify only the newest generation wins;
- keep a call in flight across a swap and verify it completes on its assigned generation;
- verify no logs or worker messages reach stdout;
- exercise multiple test-suite configs and different working directories;
- verify cleanup closes database connections and leaves no child processes behind.

## Tradeoffs

This adds process management, IPC, and lifecycle complexity. It may also briefly keep two Payload
instances and database pools alive during an atomic swap. In return, it provides standalone stdio,
keeps the harness connection stable, survives failed reloads, and makes module freshness a process
invariant instead of a cache-invalidation heuristic.
